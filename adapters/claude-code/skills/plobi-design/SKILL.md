# /plobi-design — UI/UX 设计规范

**在 Plobi 工作流中的位置：** Phase 2 方案设计 → 第一步（在 `/plobi-synthesize` 之前）

**核心职责：** 通过追问和参考收集，确定产品的视觉风格和交互规范，产出 `docs/DESIGN.md`，作为后续所有 UI 实现的强制约束文档。

---

## 触发条件

以下任意一种情况立即启动：
- 用户说"帮我设计 UI"/"我想要好看的界面"/"我在意交互体验"
- Phase 2 方案设计阶段开始
- 用户提供了截图、参考 App 名称、或视觉关键词
- 用户说"像 XXX 那种风格"

---

## 执行流程

### Step 1：收集参考（优先于追问）

先问用户有没有参考，再开始追问细节。

```
在我们定设计规范之前，你有没有：
① 截图——直接发给我，我来分析风格
② 参考 App——说名字就行（比如"像 Linear"/"像 Notion"）
③ 关键词——随便说（比如"深色、圆角大、简洁"）

没有也完全没关系，我会根据你的产品类型推荐风格让你选。
```

**如果用户有截图：** 分析截图的配色、字体风格、间距感、圆角大小、动效风格，提炼成设计语言，向用户确认。

**如果用户有参考 App：** 描述该 App 的核心设计特征，问用户"是这个感觉吗？有什么想调整的？"

**如果用户没有参考：** 根据产品类型推荐 2-3 个风格方向让用户选（见下方"风格推荐参考"）。

---

### Step 2：逐项追问设计偏好

每次只问一个维度，等用户确认后再问下一个。

**追问维度（按顺序）：**

**① 整体风格**
```
你想要哪种整体感觉？
A. 简洁极简（大量留白，字体主导，像 Linear / Notion）
B. 活泼现代（有颜色，有渐变，像 Figma / Vercel）
C. 专业严肃（深色系，数据感，像 GitHub / Raycast）
D. 你来描述
```

**② 配色**
```
配色偏好？
A. 深色模式为主
B. 浅色模式为主
C. 跟随系统
主色调有倾向吗？（比如"蓝色系"/"不要太花哨"/"品牌色是 XXX"）
```

**③ 圆角和间距**
```
界面元素的圆角感觉？
A. 大圆角（更柔和、现代感）
B. 小圆角（更严肃、精准感）
C. 直角（极简、专业）
```

**④ 动效偏好**
```
界面动画？
A. 丰富流畅（页面切换有动效，元素有弹簧感）
B. 轻微动效（只有必要的过渡）
C. 无动效（纯粹、快速）
```

**⑤ 交互反馈**
```
按钮和操作的反馈感？
A. 明显反馈（点击有缩放、颜色变化明显）
B. 轻微反馈（subtle，专业感）
```

---

### Step 3：推断和补全

用户确认风格方向后，AI 自动推断剩余细节，无需逐一追问：
- 字体推荐（根据风格方向选 1-2 个）
- 主色/辅色/背景色/文字色的具体色值
- 间距单位（4px 体系 / 8px 体系）
- 圆角大小（具体数值）
- 阴影风格（有 / 无 / 轻微）
- 图标风格（线条 / 填充 / 混合）

把这些补全后的方案给用户确认：
```
根据你选的方向，我推断以下细节，你确认一下：

主色：#2C5F8A（深蓝）
背景色：#0F1117（深色模式主背景）
文字色：#F4F4F5
圆角：8px（中圆角）
字体：Inter（界面）/ JetBrains Mono（代码区域）
间距：8px 倍数体系
动效：150ms ease-out 过渡

有需要调整的吗？
```

---

### Step 4：写入 docs/DESIGN.md

用户确认后，生成 `docs/DESIGN.md`，格式如下：

```markdown
# DESIGN.md — UI/UX 设计规范
> 本文档由 /plobi-design 生成，是所有 UI 实现的强制参考。
> Phase 3 中，AI 实现任何界面前必须读取本文档。

## 整体风格
[一句话描述风格定位，例如：专业深色系，极简布局，轻微动效]

## 配色系统
- 主色：[色值] — 用于主按钮、链接、强调
- 辅色：[色值] — 用于次级操作
- 背景主色：[色值]
- 背景次色：[色值] — 卡片、侧边栏
- 文字主色：[色值]
- 文字次色：[色值] — 描述性文字、占位符
- 危险色：[色值] — 删除、错误状态
- 成功色：[色值] — 完成、成功状态

## 字体
- 界面字体：[字体名]，尺寸体系：12/14/16/20/24/32px
- 代码字体：[字体名]（如有代码展示场景）
- 字重：Regular(400) / Medium(500) / Semibold(600) / Bold(700)

## 间距与圆角
- 间距单位：[N]px 倍数（基础单位 [N]px）
- 圆角：小组件 [N]px / 卡片 [N]px / 弹窗 [N]px

## 阴影
- 卡片阴影：[具体 CSS 值或描述]
- 弹窗阴影：[具体 CSS 值或描述]

## 动效规范
- 页面切换：[时长] [缓动函数]
- 元素出现：[时长] [缓动函数]
- 交互反馈：[时长] [缓动函数]
- 禁止：[超过 Xms 的动效 / 避免的动效类型]

## 图标规范
- 图标库：[推荐图标库，如 Lucide / Heroicons / Phosphor]
- 风格：[线条 / 填充 / 混合]
- 尺寸：16px（小）/ 20px（默认）/ 24px（大）

## 核心组件约定

### 按钮
- 主按钮：[颜色] 背景，[颜色] 文字，[N]px 圆角，hover 时 [变化]
- 次按钮：边框样式，hover 时 [变化]
- 危险按钮：[颜色] 背景，仅用于不可逆操作（删除、清空）

### 卡片
- 背景：[颜色]，圆角：[N]px，内边距：[N]px
- hover 状态：[变化描述]

### 表单输入
- 边框：[颜色]，聚焦时：[变化]
- 错误状态：[颜色] 边框 + 下方错误文字

### 空状态
- 统一使用：居中图标 + 简短说明文字 + 可选的操作按钮

### 加载状态
- 方式：[骨架屏 / 加载圈 / 淡入] — 统一使用，不混用

## 禁止事项
- 禁止在未经用户确认的情况下修改本文档中的任何设计决策
- 禁止使用本文档未定义的颜色（尤其是随机选色）
- 禁止混用不同风格的图标
- 禁止超过规范时长的动效
```

---

### Step 5：告知后续引用规则

写入完成后告知用户：

```
DESIGN.md 已写入 docs/ 目录。

后续规则：
- Phase 3 中，AI 实现任何 UI 前必须先读取 DESIGN.md
- 如果你发现界面风格和规范不符，直接说"这个不符合 DESIGN.md"，AI 必须修正
- 如果你想调整设计规范，说"更新 DESIGN.md"，重新走一遍追问流程
```

---

## 风格推荐参考（用户没有参考时使用）

根据产品类型推荐方向：

| 产品类型 | 推荐风格方向 | 参考 App |
|---|---|---|
| 生产力工具 / 任务管理 | 极简浅色或深色，大留白 | Linear、Notion、Things 3 |
| 数据看板 / 分析 | 深色系，数据密度高 | Grafana、Raycast |
| 消费者 App / 社交 | 活泼配色，圆角大，动效丰富 | Duolingo、Bereal |
| B2B SaaS | 专业浅色，信息密度中等 | Stripe、Vercel、Clerk |
| 创作工具 | 深色为主，专注感 | Figma、Arc、Craft |
| 电商 / 内容 | 浅色，图片友好，留白舒适 | Gumroad、Framer |

---

## 注意事项

- **关闭 caveman mode。** design 阶段需要完整自然语言沟通，design 结束后恢复。
- **用户提供截图时，优先从截图提取，不要无视截图去追问。**
- **不要在没有用户确认的情况下直接写 DESIGN.md。** 必须先让用户确认方案，再写入。
- **DESIGN.md 一旦写入，即为 Phase 3 的强制约束。** AI 实现 UI 时，每个组件都必须对照本文档，不得自行发挥。
- **如果用户在 Phase 3 说"这个不好看"：** 先判断是 DESIGN.md 定义的风格本身的问题，还是 AI 没有遵守 DESIGN.md。前者走更新流程，后者直接让 AI 按规范重做。
